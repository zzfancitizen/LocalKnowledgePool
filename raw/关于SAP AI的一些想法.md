# SAP TRM 智能工作流助手 — 产品需求文档 (PRD)

## 1. 产品概述

**产品名称**：SAP TRM 智能工作流助手 (TRM Workflow Intelligence)

**一句话描述**：基于对话交互挖掘 TRM 用户操作习惯，自动生成并部署 n8n 工作流，实现企业金融资产管理的单模块业务自动化编排。

**产品定位**：不是替代 SAP TRM，而是作为 TRM 的"智能副驾驶"——在日常协助中沉淀用户最佳实践，将重复性、规则性的资金业务操作转化为可复用、可调度的自动化工作流。

---

## 2. 背景与痛点

### 2.1 业务背景
TRM（Treasury and Risk Management）是企业金融资产的核心管理模块，涵盖：
- 交易管理（外汇、利率、衍生品等金融工具交易录入与生命周期管理）
- 估值与定价（按市值计价、摊销计算）
- 会计集成（交易过账、对冲会计）
- 风险分析（市场风险、流动性风险敞口计算）
- 合规报告（监管报表、内部管理报表）

### 2.2 当前痛点
1. **重复性操作负担重**：资金专员每周/每月需重复执行固定序列的事务（如月末估值运行 → 异常检查 → 会计过账 → 报表生成），耗费大量人工时间。
2. **操作依赖个人经验**：资深员工离职后，其积累的"操作套路"（如"先跑 TPM1，再检查 XYZ，有问题时回退做 ABC"）随之流失。
3. **跨事务上下文割裂**：TRM 内多个功能点（交易管理、现金流、估值、过账）相互依赖，但 SAP 标准功能不提供"一键串联"能力。
4. **Agent 泛化能力弱**：当前 SAP Agent 能回答单点问题，但无法理解"用户为什么在这个时间点做这件事"，缺乏业务流感知。

### 2.3 机会判断
- TRM 业务流程**规则性强、重复度高**，天然适合工作流自动化。
- 通过**对话式交互**挖掘习惯，既能获取上下文，又能建立用户信任，避免被动监控带来的隐私顾虑。
- n8n 作为轻量级编排引擎，通过 MCP Server 统一调用 SAP 能力，非常适合在**单模块内部**做业务编排，且客户保留完全自定义权。

## 3. 产品目标

| 目标维度 | 具体目标 | 衡量指标 |
|---------|---------|---------|
| **效率提升** | 将高频重复流程的人工操作时间减少 70% 以上 | 单流程平均执行时长（人工 vs 自动化） |
| **知识沉淀** | 将资深员工的操作经验转化为可复用工作流 | 沉淀的标准工作流数量 / 被复用次数 |
| **体验提升** | 让 TRM Agent 从"问答工具"升级为"流程助理" | 用户主动触发工作流的比例 |
| **风险降低** | 减少人工操作失误导致的财务数据错误 | 工作流执行错误率 vs 人工错误率 |

## 4. 用户画像

### 4.1 主要用户：企业资金管理专员 (Treasury Analyst)
- **日常工作**：在 SAP TRM 中录入交易、执行月末估值、生成监管报表、核对会计凭证。
- **痛点**：每月底都需要按固定步骤执行十几个 T-code，容易遗漏或出错。
- **期望**：能有一个助手记住我的操作流程，到点就自动帮我跑完，我只管看结果和异常。

### 4.2 次要用户：TRM 业务顾问 / 团队负责人
- **日常工作**：制定团队操作规范、培训新人、优化资金业务流程。
- **痛点**：每个人的操作习惯不一样，难以统一标准；新人上手慢。
- **期望**：能看到团队常见的工作流模式，把最佳实践固化为标准流程。

### 4.3 技术用户：SAP Basis / 开发顾问
- **日常工作**：维护 TRM 系统、处理接口问题、部署自动化工具。
- **痛点**：业务部门提的自动化需求太细碎，每个都要单独开发 ABAP 程序。
- **期望**：业务部门能自己配置工作流，IT 只需维护 MCP Server 与 SAP 的连接；n8n 侧由业务人员自主编辑。

## 5. 核心功能设计

### 5.1 对话式习惯挖掘 (Conversational Habit Mining)

**功能描述**：Agent 在日常对话中，通过自然交互识别用户的重复性操作模式，并将其结构化为"候选工作流"。

**具体机制**：

1. **被动识别**：当用户多次以相似方式请求帮助时触发。
   - 用户第 1 周："帮我跑一下 TPM1，公司代码 1000，今天到期的外汇交易"
   - 用户第 2 周："帮我跑 TPM1，公司代码 1000..."
   - Agent（第 3 周检测到模式）："我注意到你每周都会执行 TPM1 估值运行，而且参数很固定。你通常跑完之后还会做什么？"

2. **主动探询**：在关键业务节点（如月末、季末），Agent 主动询问用户的操作计划。
   - Agent："今天是月末，需要进行月末资金操作吗？"
   - 用户："是的，先跑 TPM1，然后检查估值差异，没问题就过账 TPM60"
   - Agent："记录下来了。这个流程你每月都做吗？需要我记住并以后自动执行吗？"

3. **上下文追问**：当用户描述一个多步骤操作时，Agent 追问细节以完善工作流定义。
   - 用户："我先检查未清交易，然后跑估值"
   - Agent："检查未清交易时，你主要看哪些字段？如果发现有异常交易，你会怎么处理——跳过还是手动修正后再继续？"

4. **习惯确认**：将识别到的模式总结给用户确认。
   - Agent："根据我们的对话，我发现你的月末标准流程大概是：
     1. 检查未清交易（T-code: TRTX）
     2. 执行估值运行（T-code: TPM1，参数：...）
     3. 查看估值差异清单（T-code: TPM12）
     4. 如无异常，执行会计过账（T-code: TPM60）
     这个总结准确吗？有没有遗漏的条件分支（比如有异常时的处理）？"

**输出物**：经过用户确认的"流程定义"（自然语言描述 + 关键参数 + 条件分支）。

### 5.2 智能工作流生成 (Workflow Generation)

**功能描述**：将确认后的流程定义转化为结构化的、可执行的工作流描述，并映射到具体的 SAP TRM 技术接口。

**具体机制**：

1. **业务语义理解**：LLM 将自然语言流程翻译为 TRM 领域的技术操作。
   - "检查未清交易" → `TRTX` 或自定义查询，参数映射
   - "执行估值运行" → `TPM1`，需要提取用户历史参数（公司代码、估值日期、交易类型等）
   - "会计过账" → `TPM60`，需要前置条件（估值成功且无异常）

2. **参数模板化**：将具体数值抽象为可配置参数。
   - 固定值：如公司代码 `1000`
   - 动态值：如"当前月末日期"、"T+1 工作日"
   - 用户输入：如"估值范围"（每次执行时由用户确认或修改）

3. **条件分支构建**：处理异常路径。
   - 主路径：检查 → 估值 → 差异正常 → 过账
   - 分支 1：差异异常 → 暂停并通知用户 → 用户确认后重试或跳过
   - 分支 2：无未清交易 → 跳过检查步骤，直接估值

4. **生成工作流草案**：输出标准化的工作流描述文档（YAML/JSON 格式），包含：
   - 步骤列表（Step ID、名称、SAP T-code/BAPI、输入参数、输出变量）
   - 依赖关系（哪些步骤必须串行、哪些可以并行）
   - 异常处理策略
   - 人类介入点（Human-in-the-loop，需要用户确认的关键节点）

**用户交互**：向用户展示可视化流程图（文字描述版或简易 Mermaid 图），让用户确认或调整步骤顺序、参数、分支逻辑。

### 5.3 n8n 自动化编排与部署 (n8n Orchestration)

**功能描述**：将确认后的工作流描述自动转化为 n8n 工作流 JSON，部署到客户专属的 n8n 实例，同时保留完整的可视化编辑能力，让客户能够根据实际业务需求自由调整流程。

**核心原则**：**AI 生成是起点，客户自定义是常态**。不追求"一键完美"，而是追求"快速生成可用草案 + 低门槛灵活调整"。

**具体机制**：

1. **n8n 节点映射**：每个 TRM 操作映射为 n8n 中的节点组合。
   - **SAP 操作节点**：通过 MCP Server 节点调用 SAP TRM 功能（如 `mcp-sap-trm.execute`），由 MCP Server 底层处理 RFC/BAPI/OData 调用细节
   - **数据转换节点**：使用 n8n 的 Set/Function 节点处理 MCP 返回的 JSON 数据，转换为下一步需要的参数格式
   - **条件判断节点**：使用 n8n IF/Switch 节点处理分支逻辑
   - **人类审批节点**：使用 n8n Wait 节点 + 邮件/企微/飞书通知，等待用户确认

2. **AI 生成工作流草案**：自动组装 n8n 工作流 JSON，生成逻辑清晰、节点规范、带有中文注释的初始版本。
   ```
   Trigger (Schedule: 每月最后一个工作日 20:00)
     → MCP Node: 检查未清交易 (TRTX) [AI生成，含注释]
     → IF Node: 是否存在异常交易？ [AI生成，含注释]
       → Yes: Wait Node (通知用户) → 用户确认后继续
       → No: 继续
     → MCP Node: 执行估值运行 (TPM1) [AI生成，含注释]
     → MCP Node: 获取估值差异清单 (TPM12) [AI生成，含注释]
     → IF Node: 差异是否在阈值内？ [AI生成，含注释]
       → Yes: MCP Node: 执行会计过账 (TPM60) [AI生成，含注释]
       → No: Notify Node: 发送异常报告给用户
     → Notify Node: 执行完成通知
   ```

3. **客户自定义编辑能力（关键设计）**：
   - **可视化编辑**：客户直接在 n8n 的 Editor UI 中拖拽、增删、修改任意节点，AI 生成的节点与普通手动创建的节点无差别
   - **节点注释**：AI 生成的每个节点自带中文业务注释（如"【AI生成】执行 TPM1 估值运行，参数由对话提取"），帮助客户理解原始意图
   - **参数外置**：关键业务参数（公司代码、估值日期、阈值等）统一提取为 n8n 工作流变量（Workflow Variables），客户可在 Execution 面板或节点配置中直接修改，无需改动节点结构
   - **安全锁定**：关键节点（如 MCP 调用节点）可标记为"建议保留"，客户删除时弹出提示说明该节点的业务作用
   - **版本对比**：支持查看"AI 生成的原始版本"与"客户修改后的当前版本"的差异，方便回滚或理解变更

4. **部署与调度**：
   - 一键发布到客户专属的 n8n 实例（每个客户独立部署，数据隔离）
   - 支持多种触发方式：定时触发（Cron）、事件触发（Webhook）、手动触发
   - 支持参数预填：TRM 主数据作为工作流变量注入，客户可在执行前调整

5. **执行监控与日志**：
   - 在 n8n 中查看每次执行的详细日志
   - 关键节点（如过账前）设置暂停点，等待业务人员确认
   - 执行失败时自动通知，并提供一键重试

### 5.4 工作流市场 / 最佳实践共享 (Workflow Library)

**功能描述**：（二期）允许团队内共享沉淀的工作流模板，形成 TRM 领域的最佳实践库。

**场景**：
- A 公司的"月末外汇估值流程"可以打包为模板
- B 公司可以直接复用并调整参数
- 避免每个企业从零开始配置

### 6.2 关键技术选型

| 组件 | 技术方案 | 说明 |
|-----|---------|------|
| **Agent 对话层** | 现有 TRM Agent 基础上扩展 | 复用现有对话能力，增加"模式识别"和"习惯挖掘"模块 |
| **LLM 核心** | GPT-4 / Claude / 国产大模型 | 负责语义理解、流程生成、自然语言交互 |
| **工作流存储** | PostgreSQL / SQLite + 向量数据库 | 存储用户习惯图谱、工作流模板、执行历史 |
| **工作流引擎** | n8n (Self-hosted，每客户独立) | 轻量级、可视化的工作流编排；客户可完全自定义编辑 AI 生成的流程 |
| **SAP 连接** | MCP Server（已有） | 通过 MCP Protocol 统一调用 SAP TRM 的 BAPI/RFC/OData；n8n 侧使用 MCP 节点 |
| **通知渠道** | 企微/飞书/邮件 Webhook | 人工审批节点和异常告警的触达 |

### 6.3 SAP TRM 接口映射示例（通过 MCP Server）

| 业务操作 | SAP T-code | MCP Tool 名称 | n8n 节点类型 | 输入参数示例 | 输出数据示例 |
|---------|-----------|--------------|-------------|-------------|-------------|
| 检查未清交易 | TRTX | `trm.checkOpenDeals` | MCP Node | `{ companyCode: "1000", dealType: "FX" }` | `[{ dealId, amount, currency, maturityDate }]` |
| 执行估值运行 | TPM1 | `trm.runValuation` | MCP Node | `{ companyCode, valuationDate, dealType }` | `{ jobId, status, resultSummary }` |
| 查看估值差异 | TPM12 | `trm.getValuationDiff` | MCP Node | `{ jobId, threshold }` | `[{ dealId, bookValue, marketValue, diff }]` |
| 会计过账 | TPM60 | `trm.postAccounting` | MCP Node | `{ jobId, postingDate, docType }` | `{ docNumber, fiscalYear, status }` |
| 获取交易详情 | - | `trm.getDealDetails` | MCP Node | `{ dealId }` | `{ dealType, counterparty, notional, terms }` |
| 现金流分析 | - | `trm.analyzeCashFlow` | MCP Node | `{ dealId, forecastHorizon }` | `{ cashFlows: [{ date, amount, currency }] }` |

**MCP Server 职责**：
- 接收 n8n 通过 MCP Protocol 发送的标准化请求（JSON-RPC）
- 内部将请求翻译为对应的 SAP RFC/BAPI/OData 调用
- 处理 SAP 认证、会话管理、错误转换
- 将 SAP 原始响应标准化为结构化 JSON 返回给 n8n

## 7. 数据流与交互流程

### 7.1 核心流程：从对话到自动化

```
阶段 1：习惯挖掘 (Week 1-4)
─────────────────────────────────────
用户: "帮我查一下公司代码1000的未清外汇交易"
Agent: [执行查询，记录请求模式]
...
```

### 7.2 异常处理策略

| 异常场景 | 处理策略 |
|---------|---------|
| SAP RFC 调用超时 | 重试 3 次，仍失败则暂停并通知用户 |
| 估值差异超出历史范围 | 暂停工作流，发送详细差异报告给用户决策 |
| 用户未在 24 小时内审批 | 自动取消执行，发送提醒 |
| n8n 节点执行失败 | 记录错误日志，通知 SAP Basis 和业务流程负责人 |
| SAP 会话失效 | 自动重新认证（通过 Service User），失败后转人工 |

## 8. MVP 范围 (Minimum Viable Product)

### 8.1 MVP 目标
在 4-6 周内交付一个可演示的最小可用版本，验证"对话挖掘 → 工作流生成 → n8n 执行"的核心闭环。

### 8.2 MVP 功能范围

**必须实现 (P0)**：
1. **习惯识别**：Agent 能检测到用户连续 3 次以上相同或相似的请求序列，并主动询问是否为固定流程。
2. **单一路线工作流生成**：支持生成无分支的线性工作流（如 A → B → C）。
3. **n8n 基础集成**：能将工作流发布到客户专属 n8n 实例，支持手动触发和定时触发；客户可在 n8n Editor 中自由修改。
4. **至少 2 个 TRM 场景**：选择最常见的两个场景作为 MVP 验证：
   - 场景 A：月末外汇交易估值流程（检查 → 估值 → 过账）
   - 场景 B：交易录入后的基础检查流程（录入 → 合规校验 → 状态更新）
5. **基础通知**：过账前的人工确认通过企微/飞书通知。

**争取实现 (P1)**：
1. 简单条件分支（IF 节点）。
2. 工作流执行日志在 Agent 对话中可查询。
3. 用户可在 Agent 中直接修改已部署的工作流参数。

**暂不实现 (P2+)**：
1. 团队级工作流市场 / 模板共享。
2. 复杂的多分支、循环、并行工作流。
3. 与外围系统（非 SAP）的集成。
4. 基于 SAP 标准日志的被动习惯挖掘（先专注对话式挖掘）。

### 8.3 MVP 成功标准
- 至少 3 名真实 TRM 用户通过 Agent 沉淀出 1 个个人工作流。
- 工作流端到端执行成功率 > 90%（排除 SAP 系统本身故障）。
- 用户在人工确认节点的平均响应时间 < 2 小时（说明触达有效）。

## 10. 风险与应对

### 10.1 业务风险

| 风险 | 影响 | 应对策略 |
|-----|------|---------|
| **用户不习惯被"观察"** | 对话式挖掘可能让用户感到被监视 | 强调"用户主动授权"机制：每次保存习惯前必须用户确认；提供"遗忘"功能 |
| **流程变更频繁** | TRM 业务规则或组织架构变化导致工作流失效 | 工作流版本管理；Agent 定期询问"这个流程还适用吗？"；参数化设计减少硬编码 |
| **过度自动化风险** | 金融操作自动过账可能导致合规问题 | 关键节点（如过账）强制人工确认；完整审计日志；支持一键回滚 |

### 10.2 技术风险

| 风险 | 影响 | 应对策略 |
|-----|------|---------|
| **MCP Server 调用失败** | MCP 与 SAP 连接超时或会话失效 | MCP Server 层实现断路器和连接池；n8n 侧配置重试策略；失败时通知客户检查 MCP 服务状态 |
| **LLM 理解偏差** | 将"检查"误解为错误的 T-code | 引入 TRM 领域知识库做校验；关键映射必须用户确认；不自动执行高风险操作 |
| **数据安全** | TRM 涉及敏感金融数据 | n8n 私有化部署；SAP 连接使用专用 Service User，权限最小化；数据脱敏存储 |

### 10.3 依赖风险

| 风险 | 影响 | 应对策略 |
|-----|------|---------|
| **MCP Server 未覆盖全部 TRM 功能** | 某些 TRM 操作尚未封装为 MCP Tool | 在 MCP Server 中快速补充缺失的 Tool 定义；AI 生成工作流时只使用已验证可用的 MCP Tools |
| **n8n 与 MCP Protocol 兼容性** | n8n 原生 MCP 节点功能有限 | 评估 n8n 内置 MCP 节点能力；如不足，可开发轻量级自定义节点或 HTTP Request 走 MCP HTTP Transport |

### 11.2 关键术语表

| **Human-in-the-loop** | 人在回路，关键决策点需要人工确认后再继续执行 |

## 12. 待决策事项 (Open Questions)

1. **n8n 部署架构**：为每个客户单独部署 n8n 实例，还是多租户共享？客户的 n8n 环境由谁运维（客户自运维 vs 我们托管）？
2. **MCP Server 能力边界**：当前已有的 MCP Server 覆盖了哪些 TRM 功能？缺失的 Tool（如 TPM1、TPM60）补充的排期如何？
3. **自定义编辑的边界**：客户能否随意删除 MCP 节点？是否需要保留"核心节点不可删除"的约束（如过账前的审批节点）？
4. **用户确认渠道**：企微/飞书/邮件，优先接入哪个？是否已有机器人账号？
5. **数据存储边界**：用户的操作历史和工作流定义是否允许存储在云端 LLM，还是必须本地私有化？
6. **权限模型**：谁有权创建/修改/执行工作流？是否需要区分"个人工作流"和"团队标准工作流"？

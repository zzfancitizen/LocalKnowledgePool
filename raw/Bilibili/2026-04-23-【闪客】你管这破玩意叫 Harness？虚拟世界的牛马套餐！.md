## 核心内容提炼

> 由 Claudian 于 2026-04-23 整理。

### 1. 流行的"造词"三部曲（视频认为这种划分比较牵强）
业界喜欢把 AI 协作的发展划分为三个阶段，本质上是不断"造新词"来体现工作量：
- **Prompt Engineering**：早期模型能力弱，需要研究各种提示词技巧、格式要求、甚至"PUA 话术"来让 AI 回答得更好。
- **Context Engineering**：模型变强后，问题不再是说话技巧，而是缺少关键上下文。于是开始研究如何给模型补充信息（RAG、工具调用、记忆压缩等）。
- **Harness Engineering**：当模型和上下文都齐备后，问题变成"AI 做事会跑偏"。就像员工每个步骤都没错，但最终结果让你想揍他。于是需要给 AI 施加**约束、权限、规范**，让它在可控范围内工作。

### 2. 更好的理解方式：两层"驾驭"
视频认为上面的三分法边界模糊，实际应该看作两个层面的**驾驭（Harness）**：
- **第一层：Agent 驾驭大模型**
  这是 Agent 本身的设计问题（如 Claude Code 的源码结构）。Agent 如何调用工具、如何规划任务、如何与大模型交互，属于这一层。
- **第二层：人驾驭 Agent**
  这才是 Harness 真正关注的重点。当 Agent 本身越来越强大时，人的工作变成了如何管理、约束、规范这个 Agent，让它长期稳定地运行下去。

### 3. Harness 的本质 = "牛马套餐" / 公司管理制度
视频用了一个非常形象的类比：
- **一个人**的公司不需要管理，靠个人能力。
- **100 人**的公司需要组织架构来管人，强调沟通协作。
- **1 万人**的公司需要规章制度、规范、约束来管组织，**稳定不出错**才是核心。

Harness 就是这个"万人公司"层面的东西——它的目的**不是发挥 AI 的最大智慧，而是把它束缚在一个可控、稳定、可替换的范围内**，让其变得标准化、"像牛马一样为其所用"，而且随时可以切换和重启。

### 4. 具体实践形式
- **个人习惯**：比如 Claude Code 窗口是临时开还是长期保持；上下文满了是压缩还是重启新窗口；让 AI 读完项目后输出一份说明文档，下次新窗口直接读文档而不是整个项目（这就是 Anthropic 提出的跨 Agent 状态交接）。
- **Spec 框架**：如 OpenSpec、Cline、Roo Code 等，本质上就是约定一套规范（提需求 → 实现需求 → 归档需求），让状态信息落地到磁盘，实现跨 Agent 的持续运转。

### 5. 技术演进的终极趋势：能力不断"内化/下沉"
视频提出两个驱动技术演进的核心规律：
1. **人类是懒惰的**：一切能写成 SOP（标准流程）的东西，最终都会变成工具和框架。
2. **功能会下沉**：一旦某个功能足够通用，就会下沉到底层成为基础能力。

因此，随着模型和 Agent 能力增强：
- 今天需要手动写的 Prompt → 将来模型内化，不需要了。
- 今天需要手动维护的跨 Agent 记忆 → Agent 内化（如 Claude Code 的记忆功能）。
- 今天需要 Spec 框架管理的流程 → 未来可能进一步内化。

**最终目标：人类一句话，AI 去执行，中间一切能力都内化成基础设施。**

### 6. 对流行公式的批判
视频不认可 **Agent = LLM + Harness** 这个公式，因为它只涵盖了"Agent 如何驾驭 LLM"这一层。实际上人类还在更高层级上驾驭 Agent、驾驭 Spec 框架，这个公式无法体现这种多层次的驾驭关系。

---

### 一句话总结
**Harness 不是某种具体技术，而是让强大但不可控的 AI 朝着你想要的方向，安全、稳定、可控地持续运转下去的"管理制度"和"约束体系"——本质上就是虚拟世界的"牛马套餐"。**
